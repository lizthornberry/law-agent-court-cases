# Court Case Transcription & Extraction Pipeline

Transcribes photographed South African Resident Magistrate's Court civil-case
files (c. 1880-1920) and extracts structured fields plus a full verbatim hearing
transcript, one JSON per case. Provider-agnostic vision-LLM backend (default
Google Gemini), incremental and resumable, designed to scale to 20,000+ images /
5,000+ cases.

## Pipeline stages

```
inventory -> classify -> segment -> transcribe -> cases -> catalog
(manifest)   (Pass A:    (group     (Pass B:       (per-case  (SQLite +
             page-type)  into cases) routed OCR)    extract)   index.json + review.csv)
```

The per-image work is **two passes** that both write into the SAME
`data/pages/<box>/<file>.json` (so `segment`/`cases` are unchanged):

1. **inventory** - walk box subfolders, natural-sort, content-hash every image to `data/manifest.json` (incremental: re-runs only flag new/changed files).
2. **classify** (Pass A) - a cheap/fast model reads `page_type`, `margin_case_number`, rotation and languages for every non-first image (NO transcription, so it's small & fast). Resumable; **live by default** (fast iteration).
3. **segment** - group pages into cases (cover detection + margin case number + box boundaries; appeals flagged) -> `data/cases.json`. Only needs Pass A.
4. **transcribe** (Pass B) - for each classified page, a model **routed by page_type** produces the verbatim transcription. Skip types (`blank`, `box_photo`) get `verbatim_text=""` with no API call. Resumable; **batch by default** (~50% cheaper, higher latency; Gemini only).
5. **cases** - per-case consolidation (cover image + page transcripts) -> one `output/cases/*.json` with all requested fields + full transcript + confidence flags. The model only extracts the **short** structured header/outcome fields; `full_transcript` is **assembled in code** by concatenating the per-page `verbatim_text` in order (verbatim, no model drift) and so is not regenerated by the LLM. Because the consolidation output is now small, this stage defaults to a cheap **flash** model (configurable knob, see below).
6. **catalog** - build `output/catalog.db` (SQLite), `output/index.json`, and `output/review.csv` (low-confidence / missing fields).

`pages` is a convenience command that runs `classify` then `transcribe`.

### Two-pass model routing (config `stages`)
A cheap model triages every page; an expensive model only transcribes pages that
need it. Configure under `stages` in `config.yaml`:

```yaml
stages:
  classify:
    model: gemini-2.5-flash          # Pass A: runs on ALL images
    # mode: live                    # optional; defaults to run.mode
  transcribe:
    mode: batch                     # Pass B default: async batch (Gemini)
    default_model: gemini-3.1-pro-preview   # Pass B fallback (the "pro" tier)
    type_models:                     # per-page_type overrides (cheaper tier)
      warrant: gemini-2.5-flash
      bill_of_costs: gemini-2.5-flash
    skip_types: [blank, box_photo]   # never transcribed (no API call)
  cases:
    provider: gemini                 # consolidation provider (default: gemini)
    model: gemini-2.5-flash          # consolidation model (default: flash)
```

### Consolidation (`cases`) stage
`full_transcript` is **assembled deterministically in code** (`consolidate.assemble_full_transcript`),
not by the model: each page's cached `verbatim_text` is concatenated **in page
order**, omitting skip-type pages (`blank`/`box_photo`) and empty pages, with a
one-line `[<filename> | <page_type>]` header per page and a blank line between
pages. This keeps the hearing transcript byte-for-byte identical to the per-page
transcription output (no LLM re-generation / drift). The model is asked only for
the short structured header/outcome fields (`case_number`, `district`,
`magistrate`, parties, `claim`, dates, appearances, `interpreter`,
`plea_verbatim`, `verdict`, plus `is_appeal`, `language_notes`,
`field_confidence`, `uncertain_fields`). `field_confidence["full_transcript"]`
is recorded as `"derived"`.

Because consolidation output is now small, the stage defaults to a cheap **flash**
model (`stages.cases`). Raise it for stronger field extraction, e.g.:

```yaml
stages:
  cases:
    provider: anthropic
    model: claude-opus-4-8           # or gemini-3.1-pro-preview, claude-sonnet-4-*
```

Default page-type -> transcribe-model mapping:

| page_type | model |
|---|---|
| `blank`, `box_photo` | *skipped* (no call, `verbatim_text=""`) |
| `warrant`, `bill_of_costs` | `gemini-2.5-flash` (flash) |
| `cover_regular`, `cover_appeal`, `plea`, `hearing`, `other` | `gemini-3.1-pro-preview` (pro, via `default_model`) |

Each `PageRecord` records both passes' state (`classified_at`/`classify_model`,
`transcribed_at`/`transcribe_model`, `transcription_status`) so each pass is
independently resumable, keyed on the image content hash.

## Setup

```bash
cd "Cursor Projects"                 # the directory that contains court_pipeline/
python3 -m venv .venv && source .venv/bin/activate
pip install -r court_pipeline/requirements.txt
cp court_pipeline/.env.example court_pipeline/.env   # then add your API key
```

Edit `court_pipeline/config.yaml` to choose `provider` and model. API keys come
from `.env` (`GOOGLE_API_KEY`, `ANTHROPIC_API_KEY`, or `OPENAI_API_KEY`).

## Usage

```bash
# 1. Inventory the images
python -m court_pipeline.run inventory

# 2. Estimate cost before any paid run
python -m court_pipeline.run estimate

# 3. Calibrate on two boxes first (classify -> segment -> transcribe -> cases)
python -m court_pipeline.run classify   --box "1-NKE 2-1-1-40" --box "1-NKE 2-1-1-11"
python -m court_pipeline.run segment    --box "1-NKE 2-1-1-40" --box "1-NKE 2-1-1-11"
python -m court_pipeline.run transcribe --box "1-NKE 2-1-1-40" --box "1-NKE 2-1-1-11"
python -m court_pipeline.run cases      --box "1-NKE 2-1-1-40" --box "1-NKE 2-1-1-11"
python -m court_pipeline.run catalog

# 4. Full run (inventory -> classify -> segment -> transcribe -> cases -> catalog)
python -m court_pipeline.run all

# 5. Later shipments: drop new box folders in, then process only new images
python -m court_pipeline.run inventory
python -m court_pipeline.run all --new-only
```

### Useful flags
- `--box NAME` (repeatable): restrict to specific boxes.
- `--limit N`: cap items processed (smoke tests).
- `--new-only`: only newly added/changed images.
- `--batch`: force async batch mode for **both** classify and transcribe (overrides per-stage defaults).
- `--force`: reprocess even if cached.

### Batch vs live (per-stage)
Each pass can set its own `mode` under `stages` in `config.yaml` (`live` or
`batch`); unset stages inherit `run.mode` (default `live`). Production defaults:
classify stays **live** (low latency while triaging); transcribe uses **batch**
(cheaper at scale). Consolidation (`cases`) is always live. Batch is **Gemini-only**;
if batch is requested but a provider lacks batch support, or batch submission
fails, the pipeline **stops with an error** (no automatic live fallback).

`transcribe` batch submits a **file-based JSONL** job per model (upload via the
files API, poll, download + map results back by `key`). The file-based path
is written to the documented google-genai batch API but **still needs
verification against a real account**; a tiny flash-model job is enough to
confirm. See the comments in `providers/gemini.py`.

## Offline test (no key, no cost)
Use the mock provider to exercise the whole flow with canned output:

```bash
python -m court_pipeline.run --config court_pipeline/config.test.yaml inventory
python -m court_pipeline.run --config court_pipeline/config.test.yaml classify   --box "1-NKE 2-1-1-22"
python -m court_pipeline.run --config court_pipeline/config.test.yaml segment     --box "1-NKE 2-1-1-22"
python -m court_pipeline.run --config court_pipeline/config.test.yaml transcribe  --box "1-NKE 2-1-1-22"
python -m court_pipeline.run --config court_pipeline/config.test.yaml cases        --box "1-NKE 2-1-1-22"
python -m court_pipeline.run --config court_pipeline/config.test.yaml catalog
```

The mock provider returns a spread of `page_type`s so one box exercises routing
(flash vs pro), skip types, and segmentation. It writes to `data_test/` /
`output_test/` (git-ignored).

### Upgrading an existing single-pass cache
Old `data/pages/**` records (pre two-pass) are auto-migrated on read: they're
treated as already classified **and** transcribed (using their existing model &
timestamp), so re-running `classify`/`transcribe` reports them as skipped instead
of re-calling the model. If you want a true two-pass re-run (e.g. to re-route
some pages to the cheaper model), use `classify --force` then `transcribe`.

## Outputs
- `output/cases/<box>__case_<number>.json` - one per case (source of truth).
- `output/catalog.db` - queryable SQLite table of all cases.
- `output/index.json` - flat index of all cases.
- `output/review.csv` - cases with missing or low-confidence fields to spot-check.

## Notes on accuracy
Cursive 1880-1920 handwriting is hard. Unreadable text is marked `[illegible]`;
each field gets a confidence rating; `review.csv` surfaces what to verify. Never
treat output as ground truth without spot-checking against the images.
